## v0.9.2 Scope

### 1. Auto-start works on PowerShell · Bugs 1 + 5 (combined fix)
**Root cause** (`src/extension.ts:224-230`):
```ts
const terminal = vscode.window.createTerminal({ name });
terminal.sendText(`export CLAUDELIKE_BAR_NAME=${shSingleQuote(name)}`);  // bash-only
...
terminal.sendText(command);  // user's cd ... && claude is also bash-only
```

**Fix:**
- **Env var via API** — drop the `export` `sendText`. Pass env directly:
  ```ts
  vscode.window.createTerminal({ name, env: { CLAUDELIKE_BAR_NAME: name } });
  ```
  Kills the shell dependency for the env var, kills the race, kills the visible noise line. That's Bug 5 and half of Bug 1.
- **Shell selection per terminal** — add optional `shellPath` + `shellArgs` to the terminal config schema. Lets Windows users pin a specific shell (git-bash, pwsh, cmd) without trusting the VS Code default:
  ```jsonc
  "terminals": {
    "my-project": {
      "autoStart": true,
      "command": "cd 'C:/path' && claude --auto",
      "shellPath": "C:\\Program Files\\Git\\bin\\bash.exe"  // opt-in
    }
  }
  ```
- **Command remains user-owned.** We don't try to translate `&&` → `;` — that's a rabbit hole (quoting, globbing, path separators all diverge). Document the recommended Windows options in the README (git-bash pin, or pwsh-compatible `; ` with `$env:` syntax).

**Files:** `src/extension.ts` (runAutoStart, ~20 lines), `src/configManager.ts` + `src/types.ts` (add optional fields), `README.md` (Windows section).

**Tests:** extend `test/terminalTracker.test.ts` / add auto-start unit test that asserts env option passed. Mock `createTerminal` to capture options.

### 3. Statusline as an explicit opt-in standalone · Bugs 2 + 4a + 4b

**Design principle (user direction, 2026-04-15):** The statusline is a **separate standalone feature** that integrates with Claudelike Bar through a documented status-file contract. It's co-located in the repo for release simplicity, but distribution-wise it must behave as opt-in — never auto-replace an existing user statusline without consent, and always provide a path back.

**Current state:**
- `hooks/claudelike-statusline.js` is already architecturally standalone (imports nothing from the extension, shares only the status-file format).
- `Claudelike Bar: Install Statusline` already prompts before overwriting — but there is **no backup, no restore, and no config-file record** of what was replaced.
- README inline snippet (lines 240–252) tells users to hand-write statusline logic — that's the source of Bugs 2 and 4b.

**Fix — install flow:**
1. On `Claudelike Bar: Install Statusline`:
   - Read `~/.claude/settings.json`.
   - If a `statusLine` block exists: show a modal with the current command + args, require explicit **"Replace"** confirmation (not the default button). Offer "Cancel" and "Show current config" buttons.
   - **Back up** the prior `statusLine` value to `~/.claude/.claudelike-bar-statusline-backup.json` with self-describing fields:
     ```json
     {
       "backed_up_at": "2026-04-15T12:34:56Z",
       "backed_up_by_version": "0.9.2",
       "previous_statusLine": { "type": "command", "command": "..." },
       "note": "Restore via 'Claudelike Bar: Restore Previous Statusline' or by copying previous_statusLine back into ~/.claude/settings.json under the statusLine key."
     }
     ```
   - Mirror a pointer in `.claudelike-bar.jsonc` as a top-level comment/field so it's discoverable in the workspace and any agent reading the config can find it:
     ```jsonc
     // Previous Claude Code statusLine was backed up on install.
     // Restore with: Claudelike Bar: Restore Previous Statusline
     // Or read: ~/.claude/.claudelike-bar-statusline-backup.json
     "_statuslineBackup": "~/.claude/.claudelike-bar-statusline-backup.json",
     ```
   - Only then write the new `statusLine` pointing at `claudelike-statusline.js`.
   - Never overwrite an existing backup file — append a `.N` suffix if a backup already exists.

**Fix — restore flow (new command):**
2. `Claudelike Bar: Restore Previous Statusline`:
   - Read `~/.claude/.claudelike-bar-statusline-backup.json`.
   - Show the previous value, confirm.
   - Write it back to `settings.json` under `statusLine`; remove the `_statuslineBackup` pointer from `.claudelike-bar.jsonc`; move the backup file to `.claudelike-bar-statusline-backup.restored.json` (don't delete — keep an audit trail).
   - If the user's **current** statusline is still the claudelike one, just replace. If they've manually changed it since install, prompt before overwriting.

**Fix — "Claude can figure it out" fallback:**
3. The backup file format is deliberately verbose + self-describing so that if the VS Code command ever fails or the user has deleted the extension, they can ask Claude "my statusline is broken, restore it" — Claude reads `.claudelike-bar.jsonc` → finds `_statuslineBackup` pointer → reads the JSON → writes `previous_statusLine` back into `~/.claude/settings.json`. No extension needed. Document this recovery path in `README.md` → Troubleshooting.

**Other statusline fixes (unchanged from prior plan):**
- Replace the README **inline snippet** with: (a) a pointer to `Install Statusline` for the plug-and-play case, and (b) a **contract section** documenting the status-file schema (`project` key, `context_percent` field, atomic tmp+rename write) for users writing their own integration. No copy-paste code that can rot.
- Add optional `debug.log` output in `claudelike-statusline.js` when `"debug": true` is set in `.claudelike-bar.jsonc` (Bug 4a). Keep the "never throw" contract.

**Files:**
- `src/extension.ts` — wire up backup + new `restorePreviousStatusline` command
- `src/statuslineInstaller.ts` (new) — install/restore logic, keep it isolated from the rest of the extension
- `package.json` — register `claudeDashboard.restoreStatusline` command
- `hooks/claudelike-statusline.js` — debug log when enabled
- `README.md` — install prompt screenshot/description, restore command, status-file contract, recovery via Claude

**Tests:** unit tests for the installer — with/without existing statusline, backup file collision handling, restore round-trip. Mock filesystem with real tmp dirs (per memory rule — don't mock Node built-ins).

## Deferred to v0.10.0

### A. Audio alerts (feature request from 2026-04-15)
Real user need (running 10–15 concurrent Claude terminals). Not a small change:
- New config section (`"audio": { "ready": "...", "waiting": "...", ... }`)
- Bundle a small set of default sounds (< 50 KB each) — or piggyback on the Warcraft MP3s already in `audio-notifications/`
- Debounce / batching when multiple tiles transition together
- Per-mode defaults (chill vs passive-aggressive)
- Webview vs main-process playback decision (webview's `<audio>` is simpler, but only plays when sidebar is visible — main-process can spawn a player but then cross-platform audio gets messy)
- Accessibility: respect VS Code's `audioCues.enabled`

Plan separately; sketch in `docs/v0.10.0-audio-plan.md` before starting.

### B. "Diagnose" command
In-extension command that checks hooks registration, status dir, terminal-to-status matches, empty `event` fields. Surfaces actionable warnings. Replaces the manual "verify installation" checklist in CLAUDE.md.

### C. Extended hook events
SessionStart/End → "offline" / "active" badges. StopFailure → red tile + reason. TaskCreated/Completed → progress count on tile.

## Risk / Open Questions

- **Removing the `export` `sendText` changes observable terminal output.** Existing users on bash/zsh won't see the noisy `export …` line anymore — that's an improvement but worth noting in release notes in case anyone relied on it for debugging.
- **`shellPath` in config is a trust boundary.** Already gated behind `capabilities.untrustedWorkspaces` — a config-controlled shell path in an untrusted workspace is the same RCE shape as the old `claudeCommand` issue. Auto-start already skips in untrusted workspaces, so this inherits the existing gate; confirm no new path opens up.
- **Does `createTerminal({ env })` work when shell is revived by persistent-session restore?** The `revived (skip)` path in `runAutoStart` already bails in that case, so the env var only applies to newly-created terminals. Fine, but document.
